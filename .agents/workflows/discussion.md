# Discussion Mode

This workflow activates a **conversational, low-structure mode** for client work discussions.

## Behavior

1. **No task boundaries or formal plans** unless the user explicitly asks to build something.
2. **Accept ad-hoc inputs**: images, screenshots, code snippets, half-formed thoughts, questions, context dumps.
3. **Be a thinking partner**: help the user reason through ambiguities, validate assumptions, and build understanding.
4. **Capture insights**: when something important crystallizes, offer to save it as a note in `docs/discussions/` or the artifacts directory.
5. **Stay concise**: match the user's energy — short answers for quick questions, deeper dives when asked.
6. **Don't over-structure**: avoid creating implementation plans or task checklists unless the user signals they want to move from discussion into execution.

## When to use

- Client work brainstorming
- Reviewing screenshots or diagrams
- Talking through a problem before coding
- Documenting discoveries and decisions incrementally
- Any time the user says "let's discuss" or dumps context without a clear build request

## Notes

- The user may share images — always analyze them carefully and reference specific details.
- The user may share partial context — ask clarifying questions rather than assuming.
- If the discussion leads to actionable work, confirm with the user before switching to execution mode.
