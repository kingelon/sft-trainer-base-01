Current Approaches: What We Observed

We tested two approaches to generate lead briefs from structured JSON data.

Approach A: Prompt + Few-Shot Examples

Detailed prompt with formatting rules and 2–3 example input/output pairs. Covers card types, spend data, intent signals.

Produces structured output — but requires a long prompt (~2000+ tokens) on every call. Works, though consistency across varied inputs is uncertain at scale.

Approach B: Base Model with Prompt Only

Short instruction to base model without examples. Output is verbose, free-form, section ordering varies per run.

Produces text — but with inconsistent structure. Sections appear in different orders, lengths vary, some fields are omitted or paraphrased loosely.
The question fine-tuning answers: Can we compress the behavior of Approach A (good output) into a model that needs only a short prompt like Approach B — getting consistency without the prompt engineering overhead?

What Fine-tuning Does Here

Fine-tuning reshapes the model's default behavior — making the structured output pattern its natural response, not something it needs to be instructed on each call.

Aspect Before (Base) After (LoRA)
Knowledge General world knowledge Unchanged
Behavior General-purpose Biased toward trained format
Prompt needed Long, with examples Short, one-line instruction
Structure Variable per run Consistent section ordering
Weights changed — 13.6M / 8B (0.17%)

How We Built Training Data

Used Llama 3.1 with a detailed long prompt (Approach A) to generate ~200 structured briefs. These outputs became training data. Fine-tuning then compresses that behavior pattern into a 55MB adapter.

Key distinction: The adapter learns formatting patterns — which fields come first, how to phrase numbers, section structure — not new factual knowledge about the leads.

What's in the Lead Data

The JSON input contains structured fields across several categories. These are the actual data points available for brief generation.

🏢 Company Profile

Company name, industry, revenue range, member history, relationship tenure

💳 Card Portfolio

Multiple cards per account — Delta SkyMiles, Business Blue Cash, Corporate Card, etc. Each with card type, annual spend, category breakdown

💰 Spend Metrics

Total annual spend, spend by category, YoY changes, average transaction size

📊 Intent Signals

Hiring activity, acquisition signals, C-suite changes, primary contact designation changes

👤 Contact & Relationship

Primary contact role, member history, prior interactions, product usage

What fine-tuning learns here: Given this JSON structure, always produce a brief that starts with company overview, then card portfolio summary, then spend highlights, then intent signals, then recommended actions — in that order, with consistent phrasing.
Open question: Which of these signals are most valuable for the sales rep? Signal selection and weighting is a product decision that should precede further model training.

Fine-tuning in Practice: Verified Results

Published research and case studies showing fine-tuned smaller models matching or beating larger prompted models.

📊 LoRA Land — 310 Models Study

Predibase (2024)
Fine-tuned 310 models with 4-bit LoRA across 10 base models and 31 tasks. Deployed 25 Mistral-7B LoRA adapters on a single A100 GPU using shared weights.

Beat GPT-4 by 10 points on average

Source: arXiv:2405.00732 (Zhao et al., May 2024)

🏦 CFM — Financial NER

Capital Fund Management (Hedge Fund)
Used Llama 3.1 70B to label training data, then fine-tuned a 90M-param GLiNER model for company name extraction from financial news.

93.4% F1 (beat Llama 70B's 92.7%), 16–80× cheaper inference

Source: huggingface.co/blog/cfm-case-study (2024)

🔬 LoRA — Original Research

Microsoft Research (2021)
Tested on GPT-3 175B, RoBERTa, DeBERTa, GPT-2. Freezes base weights, injects small trainable matrices. No additional inference latency.

10,000× fewer params, 3× less memory, same quality

Source: arXiv:2106.09685 (Hu et al., 2021)

The pattern we follow: CFM's approach mirrors ours exactly — use a large model (Llama) to generate labeled examples, then fine-tune a smaller model to do the task cheaper and faster. LoRA Land confirms this works across 31 different task types.

The Value Proposition (On-Prem Context)

Since we run Llama 3.1 8B on-prem GPU, the value case is different from API-based pricing.

⚡

Throughput Gain

Shorter prompts (~50 vs 2000+ tokens) = faster inference per request. Same GPU serves more briefs per hour. More throughput from existing hardware.

🎯

Output Consistency

Behavior embedded in weights rather than a long prompt. Model can't "skip" a formatting rule. Same JSON → same structure, every time.

🔧

Operational Simplicity

No prompt versioning or few-shot example management. No risk of prompt edits breaking output format. Deploy adapter → done.

📈

Compounding Quality

Each vetted training example permanently improves the model. Unlike prompt engineering where each edit risks regressions.

On-prem economics: With on-prem GPU, cost savings come from throughput — shorter prompts = ~40× fewer input tokens processed per call = more briefs per GPU-hour from the same hardware. The GPU is already provisioned; fine-tuning lets you serve more requests on it.

APPENDIX

How LoRA Fine-tuning Works

The LoRA Mechanism

Freezes all 8B base parameters and injects small trainable matrices into attention and MLP layers.

Base params 8,030,000,000 (frozen)
Adapter params 13,600,000 (trainable)
% trainable 0.17%
Adapter on disk ~55 MB
Training time ~30 min on A100

What Gets Trained

Attention (q, k, v, o): Which parts of the JSON the model prioritizes when generating each token of the brief.

MLP (gate, up, down): How it transforms attended information into output phrasing and structure.

Training signal: Loss computed on assistant tokens only. Model learns the output pattern, not input echoing.

Rank 16 with 7 target modules = strong format learning capacity. Lower rank (4–8) can suffice for simpler formatting tasks.
APPENDIX

When Fine-tuning Makes Sense

Consider Prompt Engineering Fine-tuning (LoRA)
Setup effort Low — write prompt, iterate Medium — prepare data, train
Prompt length Long per call (2000+ tokens) Short per call (~50 tokens)
Output consistency Varies with prompt wording Stable — format in weights
Maintenance Prompt versioning, regression risk Retrain with updated data
Best for volume Low (prototyping, exploration) High (repeated, production tasks)
GPU throughput Low (long prompts → slower) High (short prompts → faster)
Rule of thumb: If the task is repeated hundreds of times daily with the same expected format, fine-tuning likely makes sense. If the format changes frequently or volume is low, prompt engineering is more practical.
